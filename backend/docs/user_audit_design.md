# Phase 7: User Management and Audit Logging Design\n\n## 1. Overview\nThis phase focuses on implementing comprehensive User Management with role-based access control and an Audit Logging system to track all system activities for compliance and security.\n\n## 2. User Management\n\n### 2.1 User Roles and Permissions\n\n**Roles:**\n- **Director**: Full system access, approval authority, user management\n- **Accountant**: Journal entry creation, account management, reporting\n- **Sub-Accountant**: Limited journal entry and account viewing\n\n**Permissions Matrix:**\n\n| Action | Director | Accountant | Sub-Accountant |\n|--------|----------|-----------|----------------|\n| Create Journal Entry | ✓ | ✓ | ✗ |\n| Approve Journal Entry | ✓ | ✗ | ✗ |\n| View Reports | ✓ | ✓ | ✓ |\n| Manage Users | ✓ | ✗ | ✗ |\n| Manage COA | ✓ | ✓ | ✗ |\n| Manage Bank Accounts | ✓ | ✓ | ✗ |\n| Manage Vendors | ✓ | ✓ | ✗ |\n| View Audit Log | ✓ | ✓ | ✗ |\n\n### 2.2 User Model Enhancement\n\n```javascript\n{\n  name: String,\n  email: String,\n  password: String (hashed),\n  role: String (director, accountant, sub-accountant),\n  department: String,\n  isActive: Boolean,\n  lastLogin: Date,\n  loginAttempts: Number,\n  lockedUntil: Date,\n  passwordChangedAt: Date,\n  createdBy: ObjectId (User),\n  createdAt: Date,\n  updatedAt: Date,\n  deletedAt: Date,\n}\n```\n\n### 2.3 User Management Features\n- **User Creation**: Add new users with role assignment\n- **User Activation/Deactivation**: Control user access\n- **Password Management**: Reset, change, enforce complexity\n- **Login History**: Track user login attempts and times\n- **Account Lockout**: Automatic lockout after failed attempts\n- **Session Management**: Track active sessions and logout\n\n### 2.4 API Endpoints\n- `POST /api/users` - Create user\n- `GET /api/users` - List users\n- `GET /api/users/:id` - Get user details\n- `PUT /api/users/:id` - Update user\n- `DELETE /api/users/:id` - Soft delete user\n- `POST /api/users/:id/change-password` - Change password\n- `POST /api/users/:id/reset-password` - Reset password\n- `POST /api/auth/login` - User login\n- `POST /api/auth/logout` - User logout\n\n## 3. Audit Logging\n\n### 3.1 Audit Log Model\n\n```javascript\n{\n  action: String (CREATE, UPDATE, DELETE, APPROVE, REJECT),\n  entityType: String (JournalEntry, Account, User, Vendor),\n  entityId: ObjectId,\n  userId: ObjectId (User),\n  userName: String,\n  userRole: String,\n  timestamp: Date,\n  ipAddress: String,\n  userAgent: String,\n  changes: {\n    before: Object,\n    after: Object,\n  },\n  status: String (SUCCESS, FAILURE),\n  errorMessage: String,\n}\n```\n\n### 3.2 Audit Log Events\n\nThe following events should be logged:\n\n**Journal Entries:**\n- Create journal entry\n- Update journal entry\n- Approve journal entry\n- Reject journal entry\n- Delete journal entry\n\n**Chart of Accounts:**\n- Create account\n- Update account\n- Delete account\n\n**Users:**\n- Create user\n- Update user\n- Delete user\n- Change password\n- Login/Logout\n\n**Bank/Vendor:**\n- Create bank account\n- Record bank transaction\n- Create vendor\n- Record vendor invoice\n- Record vendor payment\n\n### 3.3 Audit Log Features\n- **Immutable Logging**: Audit logs cannot be modified or deleted\n- **Detailed Change Tracking**: Record before/after values for updates\n- **User Attribution**: Track which user performed each action\n- **Timestamp Precision**: Record exact time of each action\n- **IP Address Tracking**: Log IP address for security\n- **Search and Filter**: Query audit logs by date, user, action, entity\n\n### 3.4 API Endpoints\n- `GET /api/audit-logs` - List audit logs (with filters)\n- `GET /api/audit-logs/:id` - Get specific audit log\n- `GET /api/audit-logs/entity/:entityId` - Get logs for specific entity\n- `GET /api/audit-logs/user/:userId` - Get logs for specific user\n\n## 4. Implementation Plan\n\n### 4.1 Backend Implementation\n1. Enhance User model with additional fields\n2. Create Audit Log model\n3. Implement user management service\n4. Implement audit logging middleware\n5. Create API controllers and routes\n6. Implement role-based access control middleware\n\n### 4.2 Frontend Implementation\n1. Create user management pages\n2. Create audit log viewer\n3. Implement role-based UI rendering\n4. Create user profile and password change pages\n\n## 5. Security Considerations\n\n- **Password Security**: Hash passwords using bcrypt, enforce complexity\n- **Session Management**: Use JWT tokens with expiration\n- **Rate Limiting**: Limit login attempts to prevent brute force\n- **HTTPS Only**: Ensure all communication is encrypted\n- **CORS**: Configure CORS to prevent unauthorized access\n- **Input Validation**: Validate all user inputs\n- **SQL Injection Prevention**: Use parameterized queries\n\n## 6. Compliance\n\n- **Data Protection**: Comply with data protection regulations\n- **Audit Trail**: Maintain complete audit trail for compliance\n- **Access Control**: Enforce role-based access control\n- **Data Retention**: Define data retention policies\n\nThis design ensures secure user management and comprehensive audit logging for accountability and compliance.\n
